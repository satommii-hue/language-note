01

全体の構成

単一のHTMLファイル（language-note.html）にHTML・CSS・JavaScriptをすべて詰め込んだ構成。サーバーは翻訳APIのプロキシだけで、それ以外はブラウザの中だけで動いている。

HTML/CSS/JS
単一ファイル構成
フレームワーク不使用。ライブラリもゼロ。素のHTML+CSS+JavaScriptだけ。ファイルを開けばそれだけで動く。
Vercel
ホスティング + APIプロキシ
language-note.vercel.app で公開。翻訳APIはCORSの問題でブラウザから直接呼べないので、Vercelのサーバーサイド関数を中継役にしている。
GitHub
バージョン管理
satommii-hue/language-note。GitHubにpushするとVercelに自動デプロイされる。
DeepL API
翻訳エンジン
ユーザーが自分のAPIキーを持つ方式。アプリ側にキーは持たない。月50万文字まで無料。
02

データの流れ

フレーズを追加するときの流れと、翻訳するときの流れは別になっている。

フレーズ追加のとき
INPUT
日本語を入力
フォームに入力した値をJavaScriptが受け取る
↓
SAVE
localStorage に保存
JavaScriptの配列に追加して、そのままlocalStorageに書き込む
翻訳するとき
INPUT
日本語（またはいずれかの言語）を入力
↓
PROXY
Vercelのサーバー関数を経由
ブラウザからDeepLに直接リクエストするとCORSエラーになる。Vercelのサーバーが中継することで回避。
↓
API
DeepL APIに翻訳リクエスト
ユーザーのAPIキーをリクエストに含めて送る。キーはサーバーには保存されない。
↓
RESULT
翻訳結果をフォームに表示
各言語のinputに値が入る。確認してから「追加する」で保存。
CORSとは

ブラウザのセキュリティ機能。「このサイトから別のサイトへのリクエストは許可しない」というルール。DeepLのAPIは外部ドメインなので直接呼べない。サーバーサイドには制限がないので、Vercelを中継にすることで解決した。

03

ストレージの仕組み

データはすべてブラウザのlocalStorageに保存される。サーバーには何も送らない。

phrases
フレーズデータ
日本語＋各言語のテキスト＋タグをJSONとして保存。キー名：language-note-data-v1
APIキー
DeepL APIキー
このデバイスのブラウザにのみ保存。他デバイスで使うときは再入力が必要。キー名：deepl-api-key
設定
言語の表示設定・並び順
どの言語を非表示にしているか、並び順。フレーズデータと同じJSONに含まれる。
localStorageの上限 5〜10 MB
フレーズ1件 ≈ 1KB。1,000件でも1MB以下。容量を気にする必要はほぼない。
注意点

localStorageはブラウザとデバイスの組み合わせに紐づく。Chromeで保存したデータはSafariからは見えない。PCとスマホでは別々。「キャッシュをクリア」「サイトデータを削除」でも消える。→ だから定期的なエクスポートが大事。

04

使った技術・サービス

Speech Synthesis API
音声読み上げ
ブラウザ標準の機能。外部サービス不要。言語コード（bcp47）を指定して再生。英語はBritish English（en-GB）を優先して探す。
Claude API（Haiku）
スペルチェック
現在は未実装。DeepLによる自動翻訳が主な入力手段のため、スペルミスが発生しにくいと判断して削除した。
DeepL API
自動翻訳
12言語に対応。各言語のDeepLコード（例：英語はEN-GB）にマッピングして送信。
Noto Sans JP / DM Mono
フォント
Google Fontsから読み込み。日本語はNoto Sans JP、コードや言語ラベルはDM Mono。
Drag & Drop
言語の並び替え
PCはHTML5のdraggable属性、スマホはtouchイベントで実装。touchendのタイミングでDOMの最終状態をquerySelectorAllで直接読み取る方式が安定した。
05

設計の判断メモ

なぜこうなっているのか、迷ったポイントの記録。

APIキーをどこに保存するか
ユーザー各自がブラウザのlocalStorageに保存する方式を採用。サーバーにキーを持たないのでセキュリティリスクが下がる。デバイスごとに入力が必要になるが、そのほうが安全。
言語を「削除」ではなく「非表示」にしているのはなぜか
削除するとその言語のデータも失われる。非表示にするだけならデータは残る。後から「やっぱり見たい」となったときに復活できる。画面スペースの節約とデータ保全の両立。
フレームワーク（ReactやVueなど）を使わなかったのはなぜか
単一HTMLファイルで完結させる制約を守るため。フレームワークを入れるとビルド環境が必要になる。素のJavaScriptで十分な規模なので、シンプルさを優先した。
データ同期にクラウドを使わないのはなぜか
アカウント管理が不要なぶん、誰でもすぐ使えるメリットがある。JSONエクスポート→Googleドライブで十分。将来的にサーバーサイドで同期する構成も検討中。
カードのタップとロングプレスを両立させた方法
500msのタイマーで判定。touchendとmousedownが競合するのでlongPressedフラグとtouchHandledフラグを両方使って制御。スマホとPCで動作が変わらないように調整した。
06

バグ記録

発生したバグと対処の記録。

FIXED
スマホでトーストが消えない・黒い背景が残る
opacity + pointer-events: none の組み合わせで修正。visibilityやdisplayではなくopacityでフェードさせることで解決。
FIXED
カードの開閉が動かない（一度開くと閉じない）
カードのタップ（開閉）と長押し（編集モーダル）が競合していた。longPressedフラグ + touchHandled で mousedownをガード（500ms）することで修正。
FIXED
設定画面を開くたびにイベントリスナーが蓄積される
renderSettingsのたびにcloneNode(true)でコンテナをリセットする方式で解決。
FIXED
ドラッグ並び替えで順番がずれる
数値インデックスの計算をやめて、touchend時にDOMの最終状態をquerySelectorAllで直接読み取る方式に変更。
現在の状態

既知のバグはすべて修正済み。
